Today, we are going to talk about testing your code with Jest and React Testing Library.

The goal of testing is to verify that the logic you write returns the correct result.

Matchers#

For our first example, we have the transformNum function which takes a number and returns an object of different data types based on that number:

  • The number squared
  • A string with the number repeated
  • An array where the length is the number

Create the test file#

  • Create transformNum.test.js

  • First, we have to import our function for testing:

    import transformNum from "./transformNum";

  • Jest gives us some globals we can use to organize and run our tests.

  • We'll use describe to show what function we are testing

    describe("transformNum", () => {});

  • test will explain what this individual test is doing

    test("returns the correct data", () => {});

  • Here is where we are going to write our first test, so we need to call the function

    const result = transformNum(3);

Number matchers#

  • Let's talk about the number portion of the result first. What we want is a way to make to make sure that num equals 9. Or that num is less than 10.

    // result.num === 9
    // result.num < 10

  • Jest gives us a nice way to do that with the expect function and matchers

    expect(result.num).toBe(9);
    expect(result.num).toBeLessThan(10);

  • Run Tests: Pass

String matchers#

  • There are different matchers for different data types. Say we want to test the exact string value:

    expect(result.str).toBe("Number: 333");

  • Or, if we don't want to exactly match the string, we could use:

    expect(result.str).toContain("333");
    expect(result.str).toMatch(/number: 3/i);

  • Run Tests: Pass

Array matcher#

  • Next, we want to test the array value:

    expect(result.arr).toBe([0, 1, 2]);

  • Run Tests: Fail

  • Just like result.arr === [0, 1, 2] returns false, this will fail. We need to use deep equality:

    expect(result.arr).toEqual([0, 1, 2]);

  • Run Tests: Pass

Object matcher#

  • We don't have to test the object key/values separately, Jest gives us a way to test the whole result object. It uses a different matcher:

    expect(result).toStrictEqual({
    num: 9,
    str: "Number: 333",
    arr: [0, 1, 2],
    });

  • Run Tests: Pass

Bad data#

  • This function is going to throw an error if we don't pass in a number. Maybe we are okay with that, so we can write a test to verify that.

  • The structure of this expect argument is a little different, we have to use a callback so it can catch the error.

    test("no argument", () => {
    expect(() => transformNum()).toThrow();
    });

  • Maybe we don't want an error, so we can update transformNum to handle that case and then update our test.

    transformNum.js
    if (!num) return null;
    transformNum.test.js
    expect(transformNum()).toBeNull();

  • These are just a few of the many matchers available

Paths#

  • The key to testing a function well is making sure you write a test for every logic path.

  • On that note, let's take a look at getReadableNum. This function converts large numbers into a more readable format. It has 4 paths:

    • Non-numbers return empty strings
    • Numbers between 1 million and 1 billion
    • Numbers between 1 thousand and 1 million
    • Numbers under 1 thousand

  • BUT it actually has 6 paths, because removeZero has different behavior if the number ends in a .0

Tests#

  • We could write a separate test for each one of these paths, but I'm going to show you a cool iterative test instead!

  • The structure of that is a little bit different:

    test.each([])("10,100,000,000 should return 10.1B", () => {
    // test
    });

  • Each item in the array is going to be passed as an argument to this function, so we can unpack num and expected

    test.each([{ num: 10100000000, expected: "10.1B" }])(
    "10,100,000,000 should return 10.1B",
    ({ num, expected }) => {
    expect(getNiceNumber(num)).toBe(expected);
    }
    );

  • Run Tests: Pass

  • That passed so we also want to do add objects for the other paths

    test.each([
    { num: 10100000000, expected: "10.1B" },
    { num: 10000000000, expected: "10B" },
    { num: 10100000, expected: "10.1M" },
    { num: 10000000, expected: "10M" },
    { num: 10100, expected: "10.1K" },
    { num: 10000, expected: "10K" },
    { num: 999, expected: "999" },
    { num: NaN, expected: "" },
    { num: undefined, expected: "" },
    ])("10,100,000,000 should return 10.1B", ({ num, expected }) => {
    expect(getNiceNumber(num)).toBe(expected);
    });

  • Run Tests: Pass

  • But our helper text doesn't match what we are sending in, so lets use the text formatting option

    "$num should return $expected";

  • Run Tests: Pass

Mocking our own code#

  • We are going to be testing getEventStatus which returns a string indicating if an event is in the past or future. It uses the isBeforeNow function to determine which text to display.

  • Run Tests: Pass

  • When we run the tests we can see that everything is passing. And you can see that the tests run isBeforeNow.

  • Everything seems fine with this test, but after August 1st, this test will fail. We can see that if we change it to yesterday's date.

  • Run Tests: Fail

  • Since our goal is to test getEventStatus, we are going to mock isBeforeNow so we can still test both paths in getEventStatus without worrying about the date string causing this test to fail in the future.

  • Mocking isBeforeNow means we replace that function with a fake function that returns what we need.

  • To do that, we'll need to import isBeforeNow:

    import { isBeforeNow } from "./utils";

  • And we'll need to use Jest's mocking function so Jest knows to replace it with our fake function.

    jest.mock("./utils");

  • Then we'll tell the first test that isBeforeNow is going to return true because we want to test the path where the date is in the past:

    isBeforeNow.mockReturnValue(true);

  • And we'll tell the second test that isBeforeNow returns false.

    isBeforeNow.mockReturnValue(false);

  • When we run the test, we can see that the tests are passing and we are no longer calling isBeforeNow.

  • We can also verify that our date is being sent in to our mock isBeforeNow (like it would be for the real isBeforeNow).

    expect(isBeforeNow).toHaveBeenCalledWith("2021-06-01T12:00:00.000Z");

Mocking a library#

  • The last example involved us mocking our own code, but we can also mock libraries too.

  • Here we have getEventStatus which is doing the same thing as our last getEventStatus, but it uses date-fns's isBefore instead of our own function.

  • So these tests have the same problem as before, the future date will eventually fail. So we'll mock isBefore!

  • We'll import the function:

    import isBefore from "date-fns/isBefore";

  • We'll tell Jest we are mocking the function.

    jest.mock("date-fns/isBefore");

  • And we'll return true for the first test and false for the second.

    isBefore.mockReturnValue(true);

Testing components#

  • For our last example, we'll test a react component!

  • Demo the component

    • This User component displays a user's avatar and name.

    • You can edit the username by clicking the Edit button, changing the name, and then saving

  • This is what the code looks like:

    • The wrapper has a data-testid attribute which will come in handy
    • The image has an alt tag that changes with the user name
    • The username is displayed in an input element, that is disabled until the Edit button is pressed
    • Different buttons show in the edit state vs the display state
    • The save button also calls the onSave prop

Test 1: Exists#

  • We want to write tests for this component, but Jest doesn't know what to do with React.

  • This is where React Testing Library comes in! It will render the components and give us a way to query the DOM nodes for testing.

  • First thing we need to do is import react and react testing library:

    import React from "react";
    import { render, screen } from "@testing-library/react";

  • We'll use RTL's render to render the component.

    render(<User user={user} />);

  • First thing we want to check is that component renders if we provide user data. The easiest way to do that is to ask, "Is the section element present in the DOM?"

  • RTL provides several ways to query the dom. For the section element, we'll use the data-testid attribute. We can see the DOM we've selected with screen.debug.

    const section = screen.getByTestId("user");
    screen.debug(section);

  • We just want to verify that it exists, so we'll use:

    expect(section).toBeInTheDocument();

  • Run Tests: Pass

Test 2: Img#

  • Now that we know it's rendering, we want to dive into the component to make sure the individual sections are correct.

  • We'll start with the image. We want to make sure it's using the correct alt text.

  • There are a couple of options for targeting the image element, we'll start with role

    const imgByRole = screen.getByRole("img");
    // screen.debug(imgByRole);
    expect(imgByRole).toHaveAccessibleName("First Last avatar");

  • RTL provides another query that we could use here instead of this, getByAltText

    render(<User user={user} />);
    const imgByAlt = screen.getByAltText("First Last avatar");
    expect(imgByAlt).toBeInTheDocument();

  • You don't need both of these for this test, I just wanted to show you querying options.

  • Run Tests: Pass

Test 3: Input#

  • For the input, let's make sure that it displays the username properly

    render(<User user={user} />);
    const inputByRole = screen.getByRole("textbox");
    expect(inputByRole).toHaveDisplayValue("First Last");

  • There are actually a ton of other queries that you could use here instead:

    const inputByLabel = screen.getByLabelText("Name:");
    expect(inputByLabel).toHaveDisplayValue("First Last");
    const inputByPlaceholder = screen.getByPlaceholderText("Enter your name");
    expect(inputByPlaceholder).toHaveDisplayValue("First Last");
    const inputByDisplayValue = screen.getByDisplayValue("First Last");
    expect(inputByDisplayValue).toBeInTheDocument();
    const inputByTitle = screen.getByTitle("User name");
    expect(inputByTitle).toHaveDisplayValue("First Last");

  • Run Tests: Pass

Test 4: Button#

  • Let's make sure the button has the correct text for the display state

    render(<User user={user} />);
    const buttonByRole = screen.getByRole("button");
    expect(buttonByRole).toHaveTextContent("Edit");

  • Or, we could query by the text:

    const button = screen.getByText("Edit");
    expect(button).toBeInTheDocument();

  • Run Tests: Pass

Test 5: Edit#

  • Let's write a test to determine if the editing capability works! We'll render the component and pass in the onSave function.

    const onSave = () => {};
    render(<User onSave={onSave} user={user} />);
    const input = screen.getByRole("textbox");

  • In the display state, we expect the input to be disabled

    expect(input).toBeDisabled();

  • Run Tests: Pass

  • We want to fake a click on the Edit button, so we'll import RTL's user event library:

    import userEvent from "@testing-library/user-event";
    // ...
    userEvent.click(screen.getByText("Edit"));
    expect(input).toBeEnabled();

  • Run Tests: Pass

  • Then we are going to clear the input and type in a new name and hit save. We want to verify that the input has the new text:

    userEvent.clear(input);
    userEvent.type(input, "New Name");
    userEvent.click(screen.getByText("Save"));
    expect(input).toHaveDisplayValue("New Name");

  • Run Tests: Pass

  • We also want to verify that the onSave function was called with the new name, but we can't do that with the empty function we passed in. Luckly, Jest provides us with a mock function that we can monitor:

    const onSave = jest.fn();
    // ...
    expect(onSave).toHaveBeenCalledWith("New Name");

  • Run Tests: Pass

Test 6: Cancel#

  • We are actually going to repeat that whole thing, except click the cancel button instead:

    userEvent.click(screen.getByText("Cancel"));
    expect(input).toHaveDisplayValue("First Last");
    expect(onSave).not.toHaveBeenCalled();

Old#

For our last example, we'll write a test for a react component, using react testing library. RTL allows us to render our components and query the rendered component. We will use Jest to test for equality.

We have a pretty simple button component that has three states:

  • default (where the button is displayed)
  • empty (where nothing is displayed)
  • loading (where the button is diabled)

First thing we need to do is import react and react testing library:

import React from "react";
import { render, screen } from "@testing-library/react";

We'll use RTL's render to render the component.

render(<Button>Okay</Button>);

We'll use the getByRole query to test that the button is there with Jest's expect.

expect(screen.getByRole("button"));

This query will throw an error if 0 buttons are found OR more than one buttons are found, so we could leave our test like this. But let's make sure our children text is displaying properly.

expect(screen.getByRole("button")).toHaveTextContent("Okay");

If you need help debugging and what to see what RTL is rendering you can use screen.debug().

Next, we'll check the loading state:

render(<Button loading>Okay</Button>);
expect(screen.getByRole("button"));

Run the tests. This works, but we should really test the main difference, that the button is disabled.

expect(screen.getByRole("button")).toBeDisabled();

Next, we want to test the empty state.

render(<Button empty>Okay</Button>);

Since the getByRole function will throw an error when no buttons are found, we have to write the test a little differently.

expect(() => screen.getByRole("button")).toThrow();
