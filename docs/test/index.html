Testing @ Fiveable

As a team, we are aiming to achieve full test coverage for components and functions written in new code

Testing Checklist For Your PRs:#

  • Tests are readable and maintainable.
  • Test names (describe, it) are concise, explicit, and descriptive.
  • Avoid using logic within tests.
  • Don't test multiple concerns in the same test, split them into separate tests.
  • Write tests for all logic paths in a function AND edge cases.
  • Test the behavior, not the internal implementation.
  • Use a mock to simulate/stub complex class structure, methods, or async functions.

When to test#

Add tests if you:#

  • Create a new jsx component
  • Refactor an existing component
  • Create a new function
  • Update an existing util function

Don't add tests if you:#

  • Are making a small update to an existing component
  • Have already spent over hour writing tests

If you get stuck on a test for more than 30 minutes, reach out for help!#

React Testing Library#

Basic Syntax#

import React from "react";
import { render, screen } from "@fiveable-test-utils";
import Button from "./Button";
describe("Button", () => {
test("renders", () => {
render(<Button>Okay</Button>);
const btn = screen.getByRole("button");
expect(btn).toBeInTheDocument();
});
});
